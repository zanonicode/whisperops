## Approaches Explored

> In validate-and-capture mode, "approaches" documents the *path chosen* in the source plan vs. the alternatives the plan explicitly rejected. This is not a re-exploration — it's a record of the decisions already made, captured here so /define has the trade-off context.

### Approach A: Cloud-deployed kind-on-GCE-VM with active Crossplane ⭐ Recommended (and Selected by source plan)

**Description:** Single GCE VM (`e2-standard-8`) running a single-node `kind` cluster bootstrapped by `idpBuilder`. Crossplane is *active in MVP* and provisions per-agent GCP resources (bucket, SA, IAM bindings, SA key) declaratively from the Backstage template's generated CRDs. Two-tier IaC: Terraform for the cloud floor (VM, network, static IP, shared dataset bucket, bootstrap SA), Crossplane-via-ArgoCD for everything else. Three specialized A2A agents (Planner/Analyst/Writer) per Dataset Whisperer instance, each in its own namespace with dedicated GCS bucket and GCP service account.

**Pros:**
- Maximizes coverage of "modern AI Platform Engineering" technologies — exercises CNOE/idpBuilder, ArgoCD, Crossplane, kagent+A2A, Kyverno, LGTM, Langfuse, External Secrets Operator end-to-end.
- $300 GCP free credits cover the full project; spot VMs reduce cost 60-80% during dev.
- Per-agent isolation (namespace + bucket + SA) is a real production pattern, not toy.
- GitOps-everything: cloud resources are first-class git citizens via Crossplane CRDs.
- Forward-compatible: per-agent buckets already provisioned today are exactly what v0.5 (open dataset input) needs.

**Cons:**
- Single-node cluster = no HA. Pod restart = brief downtime. Documented as residual risk (§8.4).
- kind-on-VM means **no Workload Identity** — SA keys are used instead of WI bindings. Documented; in a GKE deployment Crossplane defs swap `ServiceAccountKey` for `IAMPolicyMember`.
- Crossplane adds operational surface (provider-gcp, bootstrap SA with scoped IAM, ~30-60s provisioning latency per agent).
- Two-pane observability (Langfuse + LGTM) requires conscious bridge work (Grafana Infinity panel pulling Langfuse REST aggregates).

**Why Recommended:** This is the path the source plan selected and the path being captured. The alternative comparisons below are documented to preserve trade-off context.

---

### Approach B: GKE with Workload Identity (rejected by source plan)

**Description:** Replace kind-on-VM with a GKE cluster. SA keys disappear in favor of Workload Identity bindings. CNOE reference implementation is adapted from EKS to GCP.

**Pros:**
- Workload Identity is the production-correct pattern for K8s↔GCP auth.
- Multi-node out of the box; HA is achievable.
- More representative of how the platform would actually run in prod.

**Cons:**
- "Multi-week plumbing that doesn't appear in the running platform" (source §3.2) — adapting CNOE's only-documented-on-EKS reference to GCP/GKE eats time that should go to the agent platform itself.
- Higher cost; less of the project fits in $300 free credits.
- Distracts from the project's stated learning goal: composing the *platform layer*, not re-doing CNOE plumbing.

**Why not selected:** The platform layer (Helm charts, ArgoCD apps, Crossplane definitions) is portable — it would run unchanged on GKE. The kind-on-VM choice trades a known production gap (SA keys, single-node) for faster time-to-platform-running. The gap is documented and bounded.

---

### Approach C: Single-agent monolith instead of 3-agent A2A squad (rejected by source plan)

**Description:** One Dataset Whisperer agent does plan+analyze+write in a single LLM context, using tool calls for sandbox execution.

**Pros:**
- Simpler deployment topology — one Agent CRD instead of three.
- Lower per-query A2A hop overhead.
- Less prompt surface to maintain.

**Cons:**
- Forces compromises: prompt grows long, model must be top-tier for everything, observability is coarse-grained.
- A2A protocol (kagent's native pattern) goes unexercised — defeats a core learning goal.
- Per-agent cost optimization (Planner=Haiku always, Analyst/Writer=developer-selected) is impossible.
- Each cognitive concern (routing / code-gen / communication) is genuinely distinct; collapsing them is a known anti-pattern.

**Why not selected:** The 3-agent split is justified by *cognitive distinctness*, not by ceremonial "more agents = better." Each role benefits from a different prompt and observability scope. See source §2.4 "Why this division justifies multi-agent."

## Suggested Requirements for /define

### Problem Statement (Draft)

Build a self-service internal developer platform that demonstrates a coherent solution to the operational disciplines of running LLM agents in production — cost control, prompt drift, blast-radius isolation, and LLM-aware observability — by letting any engineer ship an isolated, governed, observable Data Analyst agent over curated datasets through a 4-field Backstage form, with all infrastructure, multi-agent orchestration, policy, and cost guardrails handled by the platform.

| **Developer creating a Dataset Whisperer agent** (in-scenario) | Wants to ship a working agent without choosing vector DB, embedding model, chunking strategy, source URL, or budget — wants the platform to decide. |

### Constraints Identified

- **Budget:** Project must fit within GCP $300 free credits; SaaS dependencies (Anthropic, OpenAI, Supabase, Langfuse Cloud) must stay within their free tiers for development volume.
- **Single VM, single-node kind cluster:** Total at-idle resource budget ~9.5 GB / ~2.65 vCPU; query-burst ~12.5 GB / ~3.65 vCPU; sized into `e2-standard-8` (32 GB / 8 vCPU) with 2-3× headroom.
- **No Workload Identity:** kind has no GKE metadata service; SA keys are the substitute. Documented residual risk.
- **No HA:** Single-node cluster, single VM. Pod restart = brief downtime. Documented residual risk.
- **GCP-only:** No multi-cloud abstractions; Terraform modules and Crossplane providers are GCP-specific.
- **Greenfield build:** Working directory has no source code, IaC, or charts yet — every artifact in source §4's repo layout must be authored.
- **Working directory not yet a git repository:** `git init` is a prerequisite before /define produces git-bound artifacts.
- **Agent MCP tool surface bounded to 3 tools:** `dataset.read_profile`, `sandbox.execute_python`, `langfuse.log_observation`. Adding tools is a deliberate decision, not a casual change.
- **Form is 4 fields, not 5+:** name / description / dataset / primary LLM. Re-adding knobs is regression.
- **Dataset profile schema must be locked before agent prompts are authored:** Profile JSON is the Planner agent's primary input alongside the user question — it's a dependency edge in the build graph.
